<b>1. Introduction</b>

<b>1.1 Background Information</b>

The rapid proliferation of Internet of Things (IoT) devices and network security protocols has generated an unprecedented volume of technical documentation. From hardware datasheets and protocol specifications to security standards and architecture guidelines, domain experts frequently need to retrieve specific information from vast repositories of technical documents. Traditional keyword-based search engines often fail to capture the semantic meaning behind user queries, leading to irrelevant results and frustrated users. Moreover, general-purpose large language models (LLMs), despite their impressive capabilities, tend to hallucinate facts when queried about specialized technical domains where accuracy is paramount. This limitation is particularly severe in fields like IoT and network security, where incorrect information can have serious practical consequences ranging from hardware damage to security vulnerabilities.

Retrieval-Augmented Generation (RAG) emerged as a promising paradigm to address these challenges by combining the power of information retrieval with the generative capabilities of LLMs. Rather than relying solely on parametric knowledge stored in model weights, RAG systems retrieve relevant documents from an external knowledge base and use them as context for generation. This approach ensures that answers are grounded in authoritative sources while still leveraging the LLM's ability to synthesize and explain complex information. However, most existing RAG implementations focus on general-purpose use cases and lack the sophistication needed for specialized technical domains that require precise terminology handling and multi-stage retrieval strategies.

<b>1.2 Real-World Problem Description</b>

In professional environments dealing with IoT systems and network security, engineers and security researchers frequently encounter scenarios where they need quick answers to technical questions. Consider a security researcher analyzing a new IoT device who needs to understand the differences between MQTT and CoAP protocols, or a hardware developer trying to interpret specifications from an Arduino datasheet while designing an embedded system. Currently, these professionals must manually search through lengthy PDF documents, often skimming through hundreds of pages to find relevant sections. This process is time-consuming, error-prone, and does not scale when dealing with multiple document sources.

The challenge is compounded by the fact that IoT documentation spans diverse formats and sources—protocol RFCs, hardware datasheets, security guidelines, and architecture standards—each with its own terminology and conventions. A naive search approach that relies on exact keyword matching will fail to recognize semantically equivalent terms (e.g., "sensor" vs. "actuator"), while a purely semantic approach may miss critical exact matches for protocol names, version numbers, or technical parameters. Furthermore, without proper confidence assessment, a RAG system might generate plausible-sounding but incorrect answers when the retrieved context does not fully support the generated response—a dangerous failure mode in security-critical applications.

<b>1.3 Problem Statement and Motivation</b>

The core problem addressed by this project is the lack of an accurate, reliable, and domain-specific question-answering system for IoT and network security documentation that can:

(i) Effectively retrieve relevant information from multiple technical document sources using both keyword-based and semantic search approaches,

(ii) Accurately rank and refine retrieved results to ensure only the most pertinent information is used for answer generation,

(iii) Assess the confidence of retrieved results and generate appropriately calibrated responses, refusing to answer when insufficient context is available,

(iv) Operate entirely locally without depending on cloud-based API services, ensuring data privacy and offline capability.

The motivation for this project stems from the need to make technical documentation more accessible and queryable without sacrificing accuracy. In domains where precision is non-negotiable, any QA system must be designed with safety guardrails that prevent it from generating unverified information. By implementing a multi-stage retrieval pipeline with hybrid search, cross-encoder reranking, and tiered confidence evaluation, this project aims to demonstrate that domain-specific RAG systems can achieve production-grade reliability while remaining accessible and cost-effective through local deployment.

<b>1.4 Overview of Technologies Used</b>

The RAG-Cratoss system leverages a carefully selected stack of technologies to achieve its objectives. For document processing, PyPDF is used for reliable PDF extraction with metadata enrichment. Text chunking is performed using LangChain's RecursiveCharacterTextSplitter, which intelligently respects semantic boundaries during splitting. The embedding layer utilizes the all-MiniLM-L6-v2 model from Sentence Transformers, which produces compact 384-dimensional embeddings optimized for semantic similarity search.

For vector storage and retrieval, ChromaDB provides a persistent, locally-hosted vector database with efficient cosine similarity search capabilities. The hybrid retrieval module combines BM25Okapi from the rank_bm25 library with semantic search, fusing results using Reciprocal Rank Fusion (RRF). Result refinement is handled by the ms-marco-MiniLM-L-6-v2 cross-encoder model, which performs joint scoring of query-document pairs for improved precision.

The generation backbone uses Meta's Llama 3.2 (3B parameters) running locally via the Ollama runtime, eliminating dependencies on external API services. This local deployment approach ensures complete data privacy, reduces operational costs, and enables GPU-accelerated inference on compatible hardware. The entire system is orchestrated through Python, with modular component design allowing each phase—ingestion, embedding, retrieval, reranking, and generation—to operate independently or as an integrated pipeline.
