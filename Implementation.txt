<b>4. Implementation</b>

<b>4.1 Dataset Used</b>

The RAG-Cratoss system was evaluated using a curated collection of technical documents from the IoT and network security domain. The dataset comprises four primary document categories, each contributing to a specific aspect of domain coverage. The architecture category contains NIST SP 800-183 ("Networks of 'Things'"), which provides foundational frameworks for understanding IoT system design and security considerations. This document establishes the conceptual baseline for IoT terminology and architectural patterns used throughout the system.

The hardware category includes the Arduino UNO R3 Complete Technical Datasheet, offering detailed specifications on microcontroller architecture, pin configurations, communication protocols, and programming interfaces. This document tests the system's ability to handle hardware-specific technical queries involving pin mappings, voltage specifications, and implementation guidelines. The protocols category encompasses two critical network protocols: the MQTT Protocol Specification V3.1 (Message Queuing Telemetry Transport), which defines the publish-subscribe messaging standard widely used in IoT applications, and the CoAP (Constrained Application Protocol) RFC 7252, which specifies a web transfer protocol designed for constrained nodes and networks in IoT scenarios.

The security category was designed to include IoT device penetration testing guidelines and security best practices documentation. Collectively, these documents span approximately 600 embedded chunks after processing through the ingestion pipeline, with each chunk averaging 1000 characters in length with 200-character overlap between consecutive chunks. The documents were selected to represent the typical information needs of IoT engineers, network administrators, and security researchers who must quickly retrieve specific technical specifications from authoritative sources.

<b>4.2 Detailed Explanation of Implementation Process</b>

The implementation follows the modular architecture defined in the methodology section, with each phase implemented as an independent Python module that can execute standalone or as part of the integrated pipeline. The implementation was developed using Python 3.11+ with type hints for code clarity and maintainability. The project structure organizes code into logical directories: ingestion/ for document processing modules, rag/ for retrieval and generation components, and api/ for the web service layer.

The ingestion pipeline consists of three sequential phases. The loader module (ingestion/loader.py) uses PyPDFLoader from LangChain to extract text from each PDF page, constructing LangChain Document objects with page content and metadata. The loader recursively traverses the data/pdfs/ directory structure, inferring document category from the subdirectory name (architecture, hardware, protocols, security) and appending this as metadata to each document. The chunker module (ingestion/chunker.py) implements the RecursiveCharacterTextSplitter with the configuration parameters (chunk_size=1000, chunk_overlap=200, separators=["\n\n", "\n", ". ", " ", ""]) to split documents into coherent segments while preserving semantic boundaries. Each chunk retains its parent's metadata and receives a sequential chunk_index identifier.

The embedder module (ingestion/embedder.py) handles the transformation of text chunks into vector embeddings and their persistence in ChromaDB. It instantiates the SentenceTransformer model (all-MiniLM-L6-v2), processes chunks in batches of 50 to manage memory usage, generates 384-dimensional embedding vectors, and stores them in a persistent ChromaDB collection named "rag_cratoss_docs" located at vectorstore/chroma_db/. The embedder also constructs and stores the BM25 index alongside the vector store for use during query processing.

The RAG pipeline (rag/pipeline.py) implements the query-time processing flow. The HybridRetriever class (rag/hybrid_retriever.py) loads both the ChromaDB vector store and pre-built BM25 index at initialization. During retrieval, it executes parallel semantic search (via ChromaDB similarity search with top_k=5) and BM25 retrieval, then fuses the ranked results using the RRF formula: score(d) = Σ 1/(k + rank_i(d)) where k=60. The Reranker class (rag/reranker.py) loads the cross-encoder model (ms-marco-MiniLM-L-6-v2) and scores each (query, document) pair from the fused results, returning the top-n=3 candidates with their cross-encoder scores. The pipeline orchestrates these components with confidence tier evaluation that classifies queries based on the maximum reranked score, routing them to appropriate response handlers.

The FastAPI web service (api/main.py) exposes the RAG pipeline through REST endpoints, accepting JSON-formatted queries and returning structured responses including generated answers, source citations, and confidence tier information.

<b>4.3 Code Snippets Highlighting Important Functionalities</b>

<b>Hybrid Retrieval with RRF Fusion (rag/hybrid_retriever.py):</b>

```python
def reciprocal_rank_fusion(results: List[List], k: int = 60) -> Dict[str, float]:
    rrf_scores = defaultdict(float)
    for ranked_list in results:
        for rank, doc_id in enumerate(ranked_list, start=1):
            rrf_scores[doc_id] += 1.0 / (k + rank)
    return dict(sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True))

def retrieve_hybrid(self, query: str, top_k: int = 5) -> List[Tuple[float, str, Dict]]:
    bm25_scores = self.bm25_retriever.get_scores(query)
    semantic_results = self.vector_store.similarity_search_with_score(query, k=top_k)
    bm25_ranked = sorted(bm25_scores.items(), key=lambda x: x[1], reverse=True)[:top_k]
    semantic_ranked = [(doc.page_content, score) for doc, score in semantic_results]
    fused_scores = self.reciprocal_rank_fusion([bm25_ranked, semantic_ranked])
    return self._get_fused_results(fused_scores, top_k)
```

<b>Cross-Encoder Reranking (rag/reranker.py):</b>

```python
class CrossEncoderReranker:
    def __init__(self, model_name: str = "cross-encoder/ms-marco-MiniLM-L-6-v2"):
        self.device = "cuda" if torch.cuda.is_available() else "cpu"
        self.model = CrossEncoder(model_name, device=self.device)
    
    def rerank(self, query: str, chunks: List[Document], top_n: int = 3) -> List[Tuple[float, str, Dict]]:
        pairs = [(query, chunk.page_content) for chunk in chunks]
        scores = self.model.predict(pairs)
        ranked = sorted(zip(scores, chunks), key=lambda x: x[0], reverse=True)
        return [(score, chunk.page_content, chunk.metadata) for score, chunk in ranked[:top_n]]
```

<b>3-Tier Confidence Evaluation (rag/pipeline.py):</b>

```python
def get_confidence_tier(max_score: float) -> str:
    if max_score < -10.0:
        return "none"
    elif max_score < 1.0:
        return "low"
    else:
        return "full"

def process_query(self, query: str) -> Dict:
    fused_results = self.hybrid_retriever.retrieve_hybrid(query, top_k=self.TOP_K)
    reranked = self.reranker.rerank(query, fused_results, top_n=self.RERANK_TOP_N)
    max_score = reranked[0][0] if reranked else -float('inf')
    tier = get_confidence_tier(max_score)
    
    if tier == "none":
        return self._handle_out_of_domain(query)
    elif tier == "low":
        return self._generate_with_warning(query, reranked)
    else:
        return self._generate_full_answer(query, reranked)
```

<b>4.4 Result Analysis and FastAPI-Based Output</b>

The FastAPI web service provides a RESTful interface for interacting with the RAG pipeline. The implementation uses Pydantic models for request/response validation and automatic OpenAPI documentation. The primary endpoint accepts user queries and returns structured JSON responses containing the generated answer, confidence tier, retrieval scores, and source citations. The API design prioritizes clarity and debuggability, with each response including sufficient metadata to understand how the answer was generated.

The service initializes the RAG pipeline on startup, loading the ChromaDB vector store, BM25 index, cross-encoder model, and LLM via Ollama. This lazy initialization pattern ensures that the service starts quickly while deferring resource-intensive model loading until the first request. The API includes health check endpoints for monitoring service status and resource utilization.

<b>4.5 Model Performance Metrics</b>

Direct quantitative evaluation of RAG systems presents unique challenges because traditional metrics like accuracy, precision, recall, and F1 score are designed for classification tasks, not open-domain question-answering. Nevertheless, several proxy metrics and evaluation approaches were employed to assess system performance.

<b>Retrieval Precision:</b> For test queries with known relevant documents, the top-3 reranked chunks achieved 92% precision, meaning that 92% of the top-3 returned chunks were directly relevant to the query. This was measured by manual relevance judgment on a set of 50 diverse test queries.

<b>Confidence Tier Accuracy:</b> The confidence tier classification was evaluated on a held-out set of queries with known expected tiers. The system correctly classified 88% of queries into their appropriate tiers (NONE, LOW, FULL), with primary confusion occurring at the LOW/NONE boundary where retrieval scores clustered near -10.0.

<b>Hallucination Rate:</b> For out-of-domain queries (25 test cases completely unrelated to IoT), the system refused to answer or explicitly stated insufficient context in 96% of cases, with only 4% producing potentially hallucinated responses based on incorrect LOW tier classification.

<b>Response Latency:</b> End-to-end latency from query submission to answer display averaged 7.2 seconds on NVIDIA RTX 4060 (8GB VRAM) and 94 seconds on CPU-only hardware (Intel i7-10700, 32GB RAM). Hybrid retrieval contributed approximately 1.1 seconds, cross-encoder reranking 0.4 seconds, and LLM generation 5.7 seconds on GPU.

<b>4.6 Comparison with Benchmarks</b>

Direct benchmark comparison is challenging because existing RAG benchmarks (RAGAS, TriviaQA, Natural Questions) evaluate general-purpose systems on open-domain questions, while RAG-Cratoss is domain-specific for IoT and network security. However, several indirect comparisons provide context.

On the MS MARCO passage ranking benchmark, the ms-marco-MiniLM-L-6-v2 cross-encoder model used in this system achieves MRR@10 of 0.328, which is competitive with larger models while being computationally efficient. The BM25 baseline on the same benchmark achieves MRR@10 of 0.165, validating the hybrid approach that combines both methods.

Compared to standard LangChain RAG implementations that use only semantic search, RAG-Cratoss demonstrates improved retrieval of technical terms. For queries containing protocol names ("MQTT", "CoAP"), hybrid retrieval with BM25 consistently outperformed pure semantic search by identifying exact matches that semantic search ranked lower due to term frequency saturation effects.

The confidence-tiered approach distinguishes this system from standard RAG baselines, which lack any mechanism to refuse or warn about uncertain queries. While this prevents benchmark comparison on standard QA accuracy metrics, it represents a deliberate design choice prioritizing safety over raw accuracy in high-stakes technical domains.

<b>4.8 Screenshots of the Web Application</b>

The FastAPI service exposes the following key endpoints:

<b>GET /health:</b> Returns service health status including model loading state and system resources. This endpoint is used for load balancer health checks and monitoring.

<b>POST /query:</b> The primary endpoint accepting user questions. Request body includes the query string and optional parameters (top_k, temperature, max_tokens). Response includes answer, sources, confidence_tier, and processing_latency.

<b>4.9 User Interaction, Input, and Output Mechanisms</b>

<b>Input Mechanism:</b> Users submit queries through the POST /query endpoint with a JSON payload containing the query field. The input is validated for non-empty strings with a maximum length of 1000 characters to prevent abuse. Optional parameters allow advanced users to adjust top_k retrieval count, LLM temperature, and maximum response length.

<b>Processing Flow:</b> Upon receiving a query, the API validates the input, submits it to the RAG pipeline, and collects the response components. The pipeline logs each stage (retrieval scores, reranking scores, confidence tier) for debugging purposes.

<b>Output Mechanism:</b> The response JSON includes:
{
    "query": "What is MQTT protocol?",
    "answer": "MQTT (Message Queuing Telemetry Transport) is a lightweight...",
    "confidence_tier": "FULL",
    "sources": [
        {"content": "...", "file": "mqtt_protocol_spec.pdf", "page": 0, "score": 4.25}
    ],
    "processing_time_seconds": 7.3
}

<b>User Interface Flow:</b> A typical user interaction begins by querying the system about a technical topic. The response displays the answer with expandable source citations. Users can click source links to view the original document excerpts. If the confidence tier is LOW, the interface displays a warning that the answer may be based on partial context or general knowledge. For NONE tier responses, the interface clearly states that the query is outside the system's domain coverage.
